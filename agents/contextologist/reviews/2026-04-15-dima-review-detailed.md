## 1. Measurement Setup

### Правка 1.1 — Микроцели
**Было:** микроцели 60/120/180 сек — единственный вариант
**Не так:** микроцелей может быть много. Разные проекты требуют разных наборов.
**Надо:** агент должен предлагать клиенту варианты микроцелей (прокрутка, клики по кнопкам, просмотр планировок, время на сайте и т.д.) и согласовывать набор под конкретный проект.
**Задача Максу:** проработать и зафиксировать в skill.md расширенный список микроцелей для недвижимости с пояснением когда какую использовать.

### Правка 1.2 — Интеграция Метрика → CRM
**Было:** "настроить передачу лидов через Roistat / CoMagic"
**Не так:** Roistat и CoMagic — сторонние платные сервисы, KUBRIK их не использует.
**Надо:** KUBRIK предоставляет заказчикам кастомные дашборды собственной разработки. Интеграция Метрика → AmoCRM настраивается напрямую через API без сторонних агрегаторов.
**Куда в skill.md:** раздел Measurement Setup — убрать упоминания Roistat / CoMagic, добавить: "интеграция через кастомный дашборд KUBRIK".

## 2. Семантика

### Правка 2.1 — Бренд-кластер: что входит в бренд
**Было:** только название ЖК и домен лендинга (lm2025.ru)
**Не так:** бренд — это три независимых типа запросов:
1. Название ЖК (жк "Легенда Марусино", легенда марусино купить)
2. Основной домен застройщика (если есть жирный сайт отдельно от лендинга)
3. Название застройщика (ООО СЗ "МАРУСИНО", марусино девелопмент и т.п.)
**Надо:** перед формированием бренд-кластера агент обязан спросить: "Есть ли основной сайт застройщика отдельно от лендинга? Как называется юрлицо/бренд застройщика?"
**Куда в skill.md:** раздел Семантика → Кластер Бренд — добавить 3 типа запросов и обязательный вопрос перед генерацией.

### Правка 2.2 — Бренд-кластер: маловато семантики
**Было:** 4 ключа в бренде
**Не так:** бренд-кластер должен быть шире — все возможные написания, варианты с опечатками, с уточнениями (цены, планировки, отзывы).
**Надо:** минимум 8-12 ключей в бренд-кластере.

### Правка 2.3 — Гео в ключевых запросах: "Подмосковье" — ошибка
**Было:** "купить студию у метро Подмосковье", "студия новостройка Московская область"
**Не так:** "Подмосковье" — огромная территория, никто так не ищет. Запрос "купить студию Подмосковье" не отражает реального поведения пользователя.
**Надо:** гео задаётся через настройки кампании (радиус / полигон / конкретный город/район), а не вписывается в ключ. В ключе — конкретный топоним если он уместен: "купить студию Люберцы", "квартира у метро Котельники". Либо — вообще без гео в ключе, полагаясь на гео-настройки кампании.
**Важный нюанс:** логика зависит от города. Москва — особый случай, люди едут ради хорошего района из любой точки. Подмосковье — нет, никто специально не едет "в Люберцы" из Питера. Поэтому: для подмосковных ЖК гео-таргет важнее гео в ключе.
**Куда в skill.md:** раздел Семантика — добавить правило "гео в ключе vs гео в настройках", расписать логику для Москвы / Подмосковья / регионов.

### Правка 2.4 — Модификатор "недорого" — опасен
**Было:** "студия рядом с метро Москва недорого"
**Не так:** "недорого" привлекает аудиторию с бюджетом ниже нашего ценника. Для комфорт+ сегмента — нецелевой трафик.
**Надо:** модификаторы цены ("недорого", "дёшево", "бюджетно") — в минус-слова для сегментов выше эконом.
**Куда в skill.md:** раздел Минус-слова — добавить правило про ценовые модификаторы.

### Правка 2.5 — Длинные ключи = низкая частотность, нет данных
**Было:** агент генерирует ключи из головы без проверки частотности
**Не так:** длинные хвосты (4+ слов) могут иметь нулевую или ничтожную частотность — трафика не будет. Агент не имеет доступа к Wordstat и не знает реальных объёмов.
**Надо:** агент должен явно помечать все ключи как `assumption` (частотность не проверена) и рекомендовать прогон через Wordstat / Key Collector перед запуском. В идеале — интеграция с Wordstat MCP (в репо уже есть `mcp-servers/yandex-wordstat-mcp`).
**Куда в skill.md:** раздел Семантика — добавить обязательную пометку `[частотность не проверена, нужен Wordstat]` и рекомендацию использовать MCP.

### Правка 2.6 — Нужны более широкие запросы
**Было:** только узкие запросы с комнатностью и гео
**Не так:** широкие запросы ("семейная ипотека купить квартиру") при правильном гео-таргетинге дают объём. Не надо бояться широких ключей если гео ограничено.
**Надо:** добавлять широкие запросы по главному офферу (ипотека, сдача, цена) без обязательного гео в ключе.

### Правка 2.7 — Конкуренты: агент не запросил список
**Было:** агент написал "список не предоставлен клиентом" как будто это вина клиента
**Не так:** агент сам не запросил список конкурентов при брифинге — это его ошибка.
**Надо:** в чеклист на входе агента добавить обязательный вопрос: "Назовите 3-5 конкурентных ЖК в локации". Без этого кластер конкурентов не формируется.
**Куда в skill.md:** раздел "Что нужно на входе" — добавить "список конкурентных ЖК (обязательно запросить у клиента)".

## 4. Объявления

### Правка 4.1 — Название ЖК в объявлениях: формат обязателен
**Было:** "Легенда Марусино" (без ЖК и без кавычек)
**Не так:** название ЖК всегда пишется с указанием типа объекта и в кавычках.
**Надо:** ЖК «Легенда Марусино» — всегда, везде, без исключений.
**Куда в skill.md:** раздел Объявления — добавить правило: "название ЖК = тип объекта + кавычки + название".

### Правка 4.2 — Гео в объявлениях: конкретика, а не "у метро"
**Было:** "квартира по семейной ипотеке 3,5% — у метро"
**Не так:** "у метро" — непонятно у какого. Пользователь должен понимать локацию из объявления.
**Надо:** называть конкретный город/район ("в Люберцах", "у метро Котельники") либо выносить гео в подзаголовок/текст с конкретикой.
**Куда в skill.md:** раздел Объявления — добавить правило "гео в объявлении = конкретный топоним, не абстракция".

### Правка 4.3 — "Официальный сайт" в заголовке — не оффер
**Было:** заголовок 1 бренд-кампании: "ЖК Легенда Марусино — официальный сайт"
**Не так:** "официальный сайт" — не оффер и не УТП, не несёт смысловой нагрузки для покупателя.
**Надо:** заголовок 1 = оффер или УТП. "Официальный сайт" убрать, заменить на ключевое преимущество проекта.
**Куда в skill.md:** раздел Объявления — добавить правило: "заголовок 1 бренд-объявления = оффер/УТП, не 'официальный сайт'".

### Правка 4.4 — Слабые УТП в текстах
**Было:** "первоначальный взнос от 20%", "одобрение через Альфа-банк", "узнайте планировки онлайн"
**Не так:**
- ПВ от 20% — это условие ипотеки, не преимущество
- "Одобрение через Альфа-банк" — покупателю не важно через какой банк
- "Узнайте планировки онлайн" — слабый CTA без ценности
**Надо:** тексты объявлений = реальные УТП проекта (инфраструктура, срок сдачи, платёж в месяц, локация). Финансовые условия (банк, ПВ) — не УТП.
**Куда в skill.md:** раздел Объявления — добавить список запрещённых псевдо-УТП и примеры сильных УТП для недвижимости.

### Правка 4.5 — "Записаться на просмотр" — слабый CTA
**Было:** CTA "запишитесь на просмотр" повторяется во всех объявлениях
**Не так:** слабый и нейтральный CTA, не создаёт срочности и ценности.
**Надо:** CTA должен быть привязан к выгоде: "Узнайте условия ипотеки", "Получите планировки", "Рассчитайте платёж". "Записаться на просмотр" — допустимо только как один из вариантов.
**Куда в skill.md:** раздел Объявления — добавить примеры сильных CTA для недвижимости.

### Правка 4.6 — Сайтлинки: нужны УТП, а не навигация
**Было:** "Планировки квартир", "Семейная ипотека 3.5%", "Ход строительства", "Записаться на просмотр"
**Не так:** это навигационные ярлыки сайта — тратим рекламную площадь впустую.
**Надо:** сайтлинки = дополнительные офферы и УТП проекта. Каждая быстрая ссылка — ещё один шанс зацепить пользователя конкретной выгодой. Последний сайтлинк — контакты.
**Куда в skill.md:** раздел Объявления / Extensions — добавить правило: "сайтлинки = офферы и УТП, не навигация".

## 5. Аудитории и ретаргетинг

### Правка 5.1 — Ретаргетинг не должен показывать то же самое
**Было:** ретаргетинг = "напоминание" тем же оффером
**Не так:** если пользователь был на сайте и не сконвертировался — повторный показ того же оффера скорее всего снова не сработает.
**Надо:** ретаргетинг = другой оффер, другой угол, дополнительный стимул (специальные условия, другой формат квартиры, срочность). Либо медийная реклама на узнаваемость — но под контролем заказчика.
**Куда в skill.md:** раздел Ремаркетинг — добавить правило "ретаргетинг = другой оффер, не повторение основного".

## 6. Оптимизация

### Правка 6.1 — Decision Matrix: CTR < 3% — неправильный порядок действий
**Было:** CTR < 3% → переписать заголовок → проверить релевантность ключ-объявление
**Не так:** первый шаг не "переписать заголовок".
**Надо — правильный порядок:**
1. Мастер отчётов → вкладка "Поисковые запросы" → смотреть по каким запросам идёт трафик → нерелевантное минусовать
2. Настройки кампании → проверить ключевые запросы
3. Если трафик и ключи ок → тестировать новый заголовок (новую гипотезу)
4. Если и это не помогает → анализировать сайт (наполнение, оффер, тон, смысл — не техническую часть)
5. Если сайт ок → вопрос к продукту
**Куда в skill.md:** раздел Optimization Rules / Decision Matrix — переписать действие по сигналу CTR < порога.

### Правка 6.2 — Синхронизация с CRM: не ежемесячно, а ежедневно
**Было:** "ежемесячная синхронизация с AmoCRM"
**Не так:** ежемесячно — слишком редко. Алгоритм Яндекса обучается на сигналах, задержка снижает качество.
**Надо:** синхронизация Метрика ↔ AmoCRM — ежедневно, насколько позволяет API.
**Куда в skill.md:** раздел Optimization Rules / Ежемесячно → перенести синхронизацию с CRM в Ежедневно.

### Правка 6.3 — "Оплата за конверсию" — стратегия работает редко
**Было:** рекомендовать переход на "оплату за конверсию" при 10+ формах в неделю
**Не так:** эта стратегия работает очень редко. Низкий порог (10 форм) — недостаточно.
**Надо:** рекомендовать "оплату за конверсию" осторожно, только при стабильно высоком объёме форм (значительно выше 10/нед), с тестированием. По умолчанию — "Максимум конверсий" с недельным бюджетом.
**Куда в skill.md:** раздел Ставки — добавить предупреждение про стратегию "Оплата за конверсию".

## 7. Риски

### Правка 7.1 — "Теряем звонки" — некорректная формулировка
**Было:** "нет колтрекинга — теряем звонки как конверсию"
**Не так:** звонки не теряются — они идут. Просто не отслеживаются в рекламном кабинете.
**Надо:** если форм достаточно, кампании обучаются, менеджеры получают адекватные звонки (нет фрода — люди не говорят "я не оставлял заявку") — это терпимая ситуация. Колтрекинг подключается в первый месяц, но не является блокером запуска.
**Куда в skill.md:** раздел Risks — переформулировать риск отсутствия колтрекинга.

### Правка 7.2 — "Рынок Подмосковья" — слишком обобщённо
**Было:** "рынок Подмосковья по недвижимости перегрет"
**Не так:** Подмосковье огромное, у каждого района своя конкуренция.
**Надо:** формулировать конкретно: "рынок Люберец / Котельников / конкретного района — высокая конкуренция".
**Куда в skill.md:** раздел Risks — добавить правило конкретизации гео в описании рисков.

## 8. Минус-слова

### Правка 8.1 — Определение минус-слова: уточнить логику формирования
**Было:** "добавлять нерелевантные в минус-слова"
**Не так:** размытая формулировка. Агент должен понимать принцип: минус-слово — это слово, наличие которого в поисковом запросе запрещает показ рекламы. Добавляем не "нерелевантное слово", а то слово, при виде которого в запросе мы точно не хотим показываться.
**Надо:** переформулировать логику минус-слов: "минусуем слово, если его наличие в запросе означает, что пользователь нам не нужен" — это другая формулировка, но важная для правильного применения.
**Куда в skill.md:** раздел Семантика / Минус-слова — добавить чёткое определение и принцип формирования.
