# Эталонный процесс разработки сайта

## Этапы и артефакты

| Этап | Артефакт | Формат | Роль | Длительность |
|------|----------|--------|------|--------------|
| 1. Паспорт | Паспорт проекта | Miro / Google Doc | Проектировщик + клиент | 1 созвон + доработка |
| 2. Структура | Структура сайта | Miro (карта страниц + блоки) | Проектировщик -> внутр. ревью -> клиент | 1-2 созвона |
| 3. Прототип | Wireframes | Excalidraw / Figma | Проектировщик + клиент | ~3 недели (по UPH) |
| 4. ТЗ | Техническое задание | Google Doc | Проектировщик | ~1 неделя |
| 5. Дизайн | UI | Figma | Дизайнер + клиент | отдельный этап |
| 6. Разработка | Работающий сайт | CMS / код | Разработчик | отдельный этап |

## 1. Паспорт проекта

Фиксирует "зачем" до "как". Без паспорта структура делается вслепую.

**Что должно быть:**
- Бизнес-цели проекта (не "сайт", а "лидогенерация / расширение рынков / узнаваемость / лояльность")
- ЦА: сегменты, JTBD, боли
- Позиционирование (УТП, tone of voice)
- Анализ конкурентов (что делают, в чём у нас преимущество)
- Ограничения: CMS, бюджет, сроки, интеграции
- Бренд-элементы: лого, палитра, шрифты (если есть)

**Замечание из кейса UPH:** созвон по паспорту занимает много времени, и часто не успеваешь обсудить все спорные вопросы. Это нормально - недостающие куски фиксируются как "на обсуждение" и закрываются позже.

## 2. Структура

Карта всех страниц сайта + список блоков на каждой + переходы между страницами.

**Что должно быть:**
- Шапка: логотип, меню, контакты, price (опционально), языковые версии, поиск, кнопка "получить консультацию"
- Главная: промо + каталог + готовые решения + proof + форма
- Под-категория (если каталог): промо + продуктовая подборка + тех. детали
- Карточка товара/услуги: галерея + описание + преимущества + форма + документы + характеристики
- Внутренние страницы под бизнес-процессы (дилеры, партнёры, госзакупки, сотрудничество)
- О компании, Новости, Контакты, Политика
- Футер: дубль меню, контакты, поиск

**Методология принятия решений:**
- Если есть несколько вариантов организации каталога (по технологии / отрасли / виду) - сделать **все варианты в Miro** и обсуждать с клиентом.
- Сверить группировку каталога с **ЦРМ и 1С** клиента - иначе придётся переделывать.
- Для каждого "серого" блока фиксировать **вопрос клиенту** прямо в структуре ("здесь пусто - нужна информация: какие компании интересны").

**Внутренняя проверка перед клиентом:**
Обязательный созвон с коллегой (в UPH: Артём -> Настя). 5-10 минут walkthrough ловят 80% проблем до клиента.

## 3. Прототип

Wireframes всех ключевых экранов (главная, каталог, под-категория, карточка, отраслевая, контакты + все специальные страницы).

**В кейсе UPH:** Excalidraw, ~157 текстовых элементов, детально до уровня "поле ввода имени" / "кнопка отправить" / "tg wa".

**Цена референс:** 40-60К руб.

## 4. ТЗ

Документ для разработчика: стек, CMS, интеграции, SEO-требования, админка, формы и их назначение, интеграция с CRM, аналитика, мобильная версия, производительность.

**Цена референс:** 30-40К руб.

## 5-6. Дизайн и разработка

За рамками методологии структуры. Тут включаются дизайнер + разработчик.

## RE-режим: дополнительные этапы для B2C-недвижимости

На основе кейса [Allex Villas](case-allex-villas/). Применяется для застройщиков, агентств недвижимости, инвест-платформ.

### Когда переключаться в RE-режим
- Сайт = листинг объектов (а не каталог товаров).
- ЦА — конечный покупатель / инвестор (не B2B-закупщик).
- Ниша YMYL (Your Money Your Life) — Google требует особый E-E-A-T-режим.
- Нужны калькуляторы доходности, прогнозы, фильтры по 5 осям.

### Основные отличия флоу

| Этап | B2B-каталог (UPH) | B2C-недвижимость (Allex Villas) |
|---|---|---|
| 1. Паспорт | базовый | + бизнес-модель девелопера, инвест-инструменты, юр-структура |
| 2. Структура | каталог 3 группировками | гибридная (5 осей фильтрации × категории объектов) |
| **2b. SEO-этап** | (отсутствует) | **анализ конкурентов + семантика + анализ КФ + ТЗ блоков + гибридная структура + мета-теги** |
| 3. Прототип | wireframes | wireframes + калькулятор доходности |
| **3b. Мудборд** | (отсутствует) | **отдельный созвон по визуальному референсу** |
| 4. ТЗ | базовое | + schema.org/Article + RealEstateListing + multi-language + hreflang |
| 5. Концепты | дизайнер сразу с прототипом | **2-3 итерации дизайн-концептов от ОС-дизайнера** на основе мудборда |
| 6. Разработка | базовое | + калькуляторы + фильтры + индексация листингов |

### Этап 2b: SEO-этап (между структурой и прототипом)

**Артефакты:**
1. **Анализ конкурентов** — таблица 8–12 прямых конкурентов по 9 параметрам (Age / DR / Traffic / KW / RD / CWV / AI Visibility / Schema / E-E-A-T-комментарий). Считается медиана по нише — целевой ориентир.
2. **Семантическое ядро** — кластеры коммерческих + информационных запросов (для Allex Villas — 16 кластеров, 926 запросов).
3. **Анализ КФ (коммерческих факторов)** — что есть «у каждого второго конкурента» и должно быть у нас. По 4 типам страниц: главная / листинг / детальная карточка / блог.
4. **Гибридная структура** — финальная sitemap, объединяющая бизнес-логику и SEO-кластеры. Обычно — sheet с раскрытыми осями фильтрации.
5. **Мета-теги** — title/description-шаблоны для всех типов страниц.

**Кто делает:** SEO-специалист параллельно с проектировщиком. Проектировщик фиксирует структуру, SEO-эксперт обогащает её семантикой и блоками-КФ.

**Длительность:** 1–2 недели.

**Гейт:** клиент согласовал ТЗ на блоки, утвердил гибридную структуру, мета-теги переведены на язык(и) рынка.

### Этап 3b: Мудборд (между прототипом и концептами)

**Артефакт:** Figma-доска или Miro с референсами + 1 созвон с дизайнером + клиентом.

**Цель:** до начала отрисовки концептов согласовать **визуальную эмоцию** бренда (премиум / технологичный / семейный / экзотика / активный) и базовые элементы (палитра, шрифты, фотостиль, тон иллюстраций).

**Кто делает:** ОС-дизайнер (или арт-директор) + проектировщик.

**Длительность:** 1 созвон + доработка референсов 2–3 дня.

**Почему отдельно:** в недвижимости визуальная подача = 50% продажи. Без мудборда — 5 итераций «не то / не той эмоции», с мудбордом — 2 итерации до финала.

### Этап 5: Концепты (RE-специфика)

В отличие от B2B-флоу, где дизайнер сразу делает финальный UI после прототипа, в RE-флоу:

1. **Концепт-итерация 1** (через 2–3 недели после мудборда) — главная + 1–2 ключевых страницы.
2. **Правки** — клиент даёт фидбэк.
3. **Концепт-итерация 2** — ещё 1 ключевая страница + правки.
4. **Финальные ОС-концепты** — все ключевые экраны, готовые к разработке.

В кейсе Allex Villas: бриф на дизайн (22.10) → мудборд-1 (06.11) → мудборд-2 (20.11) → ОС-концепты (25.11). 5 недель от старта дизайна.

### Гейты для RE-режима

- **После SEO-этапа:** ТЗ на блоки утверждено? Гибридная структура согласована с клиентом? Мета-теги на нужных языках?
- **После мудборда:** клиент видит и подтверждает «визуальную эмоцию» бренда?
- **После концептов:** все ключевые экраны (главная, листинг, карточка проекта, страница услуги, страница локации, блог-статья, контакты) отрисованы?
- **После ТЗ:** schema.org-разметка прописана? Hreflang? Калькулятор рентабельности есть в ТЗ?

### Как вести демо структуры клиенту (RE)

На основе разбора созвонов Allex Villas (см. `case-allex-villas/process-from-calls.md`).

1. **Открывающий риф о смете перед стартом демо** (~60 сек). Явно проговорить, что смета сейчас в вилках; что каждое решение по блоку влияет на вилку дизайна / разработки; что вилки снимаются последовательно по этапам (после структуры -> фикс по дизайну, после прототипа -> фикс по разработке). Это снимает 80% возражений «а сколько это будет стоить» прямо в процессе обсуждения блоков.
2. **Предупредить о размере дерева заранее.** «Структура - ещё больше дерево, чем паспорт. И это нормально.» Снимает шок от объёма документа.
3. **Идти строго: правая часть Miro (шапка/главная/футер - сквозные элементы) -> левая часть (внутренние страницы).**
4. **По каждому блоку: «что есть» -> «почему именно так» -> «что я ещё придумала / что нужно от вас».** Не презентация, а walkthrough с обоснованием.
5. **Формулировки помечать как «условные».** Это разделяет дискуссию: смысловой поток правится сейчас, копирайтерский - на этапе прототипа.
6. **Все правки красным прямо в Miro во время созвона.** Не «вспомним по записи» - моментальная фиксация на схеме.
7. **«Пустоты» отмечать как пункт-1 / пункт-2 / пункт-3.** Каждый placeholder становится запросом к клиенту в чат после созвона с явным владельцем (отдел продаж / маркетолог / РОП клиента).
8. **Не пытаться пройти всё за один созвон.** «Инфы очень много... надо чуть-чуть переварить теперь.» Скинуть запись + ссылку на Miro - домашка клиента до следующего созвона.
9. **В конце - явный пересказ next steps** + переход к смете (что фиксируем, что оставляем на пересчёт, когда следующий созвон).
10. **Срывы по графику клиента (стройка / документы / РОП в отпуске) фиксировать как «блок остаётся в структуре, контент трансформируется».** Не выкидывать, а помечать «опционально включается, когда клиент готов».
11. **Технические возражения не разрешать на ходу.** Если клиент жалуется на капчу / сервера / интеграции - фиксировать в backlog с владельцем, не блокировать структурный диалог.
12. **SEO-вопросы передавать в чат к SEO-эксперту**, а не «угадывать» решение на структуре.

### Как делать бриф на дизайн (RE)

Бриф заполняется поверх готового прототипа, а не вместо него. Дизайнер видит каркас + эмоцию + анти-эмоцию.

1. **Цветовая гамма** - ссылка на бренд-гайдлайн клиента + явный список запретов (на Allex Villas: исключили жёлтый - «мы его нигде не используем»). Запреты сильнее, чем разрешения.
2. **Эмоция бренда** формулируется через ряд итераций. Не одно слово «премиум», а связка из 3-5 формулировок: спокойная уверенность / зрелость / тихая роскошь / slow living / надёжность. И **анти-эмоция** (что мы НЕ хотим): «кричащий люкс» -> избегать.
3. **Атмосфера локации** - явное требование. На Allex Villas: «настроение Бали должно считываться, чтобы он не был как очередной сайт девелопера». Решение - тёплые нейтральные цвета + текстуры (бетон) высокого качества, а не цветовое решение.
4. **Формы**: строгие края, тонкие линии, скругления минимальны (Allex Villas: «не более 5%», в большинстве случаев избегать).
5. **Минимализм** + явное требование «достаточно воздуха, но без откровенно пустых неэстетичных мест». Анти-пример - блок текущего сайта клиента, где «вверх прилипла одна текстовка, вниз другая, в середине пусто».
6. **Контент-приоритет**: смыслы / фото / тексты - что первично. Если фотобанк ограничен - явно зафиксировать «акцент на смыслы, фото подбираем под имеющийся фотобанк».
7. **Анти-паттерны (запреты), которые обязательно фиксируются в брифе RE-проекта**:
   - Никаких эмодзи (типичный кейс: остатки от ChatGPT-копирайтинга прошлой команды).
   - Минимум анимаций (тяжелят сайт, тормозят на iframe-просмотрах).
   - Никаких видео на фоне главной.
   - Не использовать монограмму бренда на каждом текстовом блоке.
   - Гигантизм (огромные шрифты на весь экран) - не подходит.
8. **Старый бриф клиента (от предыдущей команды) разбирается «правкой по строчкам».** Не выкидывать - проходимся точечно: что переносим, что выкидываем, что добавляем.
9. **Договорённость о количестве итераций**: 3 разных концепта главной -> правки -> 1-2 итерации с комбинированием -> финал. Норма - 2-3 итерации, не «попадание с первого».
10. **Ограничение скоупа**: концепт = только главная страница. После согласования главной - дизайн внутренних страниц по согласованному стилю.

### Как вести мудборд (RE)

Двухчастный формат, повторяющийся в обоих мудборд-созвонах:

1. **Часть 1 - walkthrough по 6-9 референсам.** По каждому: расшарить экран, назвать вслух «что мы смотрим» (соотношение шрифтов, расположение, переходы, способ представления контента), вытянуть комментарии клиента, повторить услышанное в виде формулировки -> клиент уточняет.
2. **Открытый вопрос -> уточнение -> формулировка.** «Что думаете?» -> «вот эта штука сразу хочется закрыть» -> «то есть не нравится сетка, акцентов нет?» -> «да, глазу не за что зацепиться».
3. **Тактика «подборка швака»**. Анти-референсы дают контраст и помогают клиенту артикулировать «как точно нет». Это не пустая трата времени - это сильный инструмент извлечения предпочтений.
4. **Сравнение «нравится здесь / не нравится здесь»** в рамках одного сайта - не «сайт хорош/плох», а «вот этот блок огонь, вот этот трэш».
5. **Озвучивание элементов вслух**: «здесь акцентные заголовки, большие шрифты, акцент на визуальной составляющей» -> просить клиента согласиться или нет. Перевод визуала в слова.
6. **Если эффект не загружается у нас** - просим клиента показать со своей стороны (расшарить экран). Не блокируем диалог.
7. **«Я могу скрин снять»**: если клиент не может объяснить, что нравится - проектировщик предлагает записать скринкаст конкретного эффекта и обсудить.
8. **Закрытие мудборда единым «спасибо, дальше - концепт».** Если что-то не зафиксировано - вынос в отдельный созвон до старта концепта, не переносим в концепт «висящие» вопросы.
9. **На втором мудборде / первой итерации концептов**: показываем 2-3 концепта рядом, фиксируем «правый блок взять из чёрного, левый из белого, эту фишку из третьего». Финал = явный список блоков из каждого концепта в комбинированный.

### Как защищать концепты (RE)

1. **Концепты не презентуются walkthrough'ом** (в отличие от структуры/прототипа). Открыть файл, дать клиенту overall-впечатление, потом собрать конкретный список правок.
2. **Подведение итогов вслух после каждого блока.** «Что мы должны сделать? Первое - сочетать чёрный и белый. Второе - убрать жирный. Третье - сделать буквы меньше.» Каждое решение подытоживается.
3. **Спрос конкретики при абстрактных правках.** «Мы должны сделать ближе к чему?» -> если клиент не может словами, просим Pinterest-референсы / скрины / видео в чат.
4. **Защита SEO-блоков перед удалением.** Если клиент намекает на «выкинем» - объяснить, что блок там по требованию SEO; короче страница -> хуже SEO результат. Без жаргона.
5. **Конфликт «премиум vs продающий» решается через переоформление**, а не удаление. SEO-блоки остаются, дизайнер переоформляет их в имиджевой стилистике. Это требует дополнительной итерации.
6. **Передача решения дизайнеру, а не клиенту.** Если клиент сам говорит «это должен делать дизайнер, у него знаний больше» - принять health-границу: дизайнер решает «как», клиент даёт «что».
7. **Цвет вместо жирного шрифта** - простой переводимый guideline для дизайнера в RE-проекте, который любит «тонкие линии».
8. **Финальный гейт - не «подписать договор на разработку», а «вернуться через N дней с правками от дизайнера».** Не торопим решение даже если близко к финалу.

### Анти-паттерны RE-флоу (доп. к общим)

7. **Прыгать в концепты без мудборда.** 5 итераций «не той эмоции» против 2 итераций с мудбордом.
8. **Делать SEO после прототипа.** SEO-этап должен быть до прототипа — иначе структура и блоки будут переделываться.
9. **Игнорировать YMYL-режим.** Без блога с авторами + микроразметки + отзывов + лицензий сайт недвижимости не выйдет в топ за разумные сроки.
10. **Один листинг на все типы объектов.** Если есть и виллы, и квартиры, и отели — это 3 листинга с уникальными FAQ. Иначе каннибализация выдачи.
11. **Скрывать команду.** YMYL требует «человеческого лица» — фото, ФИО, лицензии. Без этого Google не доверяет.
12. **Калькулятор ROI как footer-блок.** Это центральный CTA для инвестора — должен быть в основной навигации и/или sticky.

## Гейт-поинты (checkpoints)

- **После паспорта:** согласован ли бизнес-смысл проекта? Есть ли измеримые KPI?
- **После структуры:** все ли страницы покрыты? Сверено ли с ЦРМ/1С? Закрыты ли "пустоты"?
- **После прототипа:** клиент "видит" сайт? Все состояния форм покрыты (отправка / успех / ошибка)?
- **После ТЗ:** разработчик понимает, что делать, без дополнительных вопросов?

## Анти-паттерны (замеченные в кейсе)

1. **Прыгать в прототип без структуры.** Получается каша и много переделок.
2. **Не сверять каталог с ЦРМ.** Клиент потом говорит "у нас в 1С по-другому" - переделка всего каталога.
3. **Показывать клиенту один вариант.** Когда есть неоднозначность - показать 2-3 и обсудить. Иначе спорим впустую.
4. **"Ничего не найдено" как тупик.** Если поиск пустой - форма обратной связи, а не 404.
5. **Price без процесса актуализации.** Устаревший прайс хуже, чем его отсутствие.
6. **Показывать текущих дилеров потенциальным.** Демотивирует: "рынок уже захвачен".
