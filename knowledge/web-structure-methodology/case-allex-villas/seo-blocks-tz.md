# ТЗ на блоки и коммерческие факторы (4 типа страниц)

Эталон ТЗ от SEO-команды на 4 ключевых типа страниц девелоперского сайта. Каждый блок проверен относительно конкурентов («у каждого второго есть»), для каждого зафиксирована позиция клиента (✅ согласовано / ❌ не делаем / ⚠️ под сомнением).

Используй как чек-лист на любом B2C-сайте недвижимости.

### Шаблон футера для девелопера
- Юр. инфо + контакты
- Меню коммерческих страниц (Услуги, Продажа, Аренда, Разработки)
- Меню информационных страниц (О компании, Блог, FAQ, Контакты)
- Ссылки на все соцсети, где идёт реклама проектов (Instagram, YouTube, Telegram, Facebook, …)

## Универсальные принципы (применять на любом проекте недвижимости)

1. **Каждый блок главной = ссылка на страницу.** Главная — дайджест, а не самостоятельный контент.
2. **Форма обратной связи минимум 2 раза на странице** (середина + низ для длинных страниц).
3. **CTA-кнопка «связаться»** в баннере + после блока с деньгами (прогноз доходности, цена) — момент максимального интереса.
4. **Перелинковка** между проектами в одной локации + между статьями блога — критична для индексации.
5. **FAQ-блок** на листингах — обязателен, если у вас несколько листингов одного типа (иначе каннибализация).
6. **Schema.org везде:** Article на блоге, Product/RealEstateListing на карточках проектов, Organization в футере, BreadcrumbList на всех страницах.
7. **Author profiles:** для блога — выделенные карточки авторов с ссылками на профили (для YMYL это must-have).
8. **Mobile-first:** всё проектируется с мобильной версии, потом масштабируется на десктоп.

## Как применять чеклист на новом клиенте

1. По каждой строке таблицы — спросить клиента: «делаем / не делаем / надо обсудить».
2. Зафиксировать решения прямо в этом доке (в правой колонке).
3. Если клиент говорит «не делаем» по блоку, который есть у каждого второго конкурента → отметить как риск в дашборде проекта.
4. Если клиент говорит «делаем» — блок попадает в прототип на следующей итерации.
