# Правила формирования отчётов на понятном человеку языке

Источник: Смарин (методология), 2026-04-29.
Применять: при формировании любых отчётов, КП и брифов для клиентов.

## Главное (преамбула + §1)

1. Отчёт пишется ДЛЯ ЗАКАЗЧИКА. Никаких технических кодов, никакой воды, никаких ссылок на параграфы методологии. Простой русский язык.
2. Каждый абзац = конкретный вывод, действие или вопрос. Тест: «прочитал – могу ли как заказчик сделать вывод/действие?». Если нет – переписать или удалить.
3. Не лить воду. «РК активна», «72 из 76 что-то меняли», «срез проверен» (без результата) – это констатация, а не вывод. Не пишем.

## Запреты на жаргон (§3, §4)

4. API-коды и enum-ы – переводить на русский по таблицам §3.1–§3.5: `TEXT_CAMPAIGN` -> «РСЯ» / «Поиск», `WB_MAXIMUM_CONVERSION_RATE` -> «Максимум конверсий с недельным бюджетом», `LSC` -> «Последний значимый переход кросс-девайс», и т.д.
5. Не упоминать API-методы: `campaigns/get`, `bidmodifiers/get`, `audiencetargets/get` – это для нас, не для заказчика.
6. Не упоминать параграфы методологии: «БЗ §2», «по методологии §X.Y», «маркер 🔵 в нашей методологии». В отчёте 🔵 – это просто «открытый вопрос», без отсылки.
7. Не выводить разделы вида «Статус 7 шагов методологии», «Шаг 0 / 1 / 2 – пройден (частично)» (§6.12). Если хочется показать, что проверено – пишем раздел «Что проверено в этом аудите» простым языком.
8. Не выводить пути к JSON-полям: `Settings.Items`, `TextCampaign.AttributionModel`, `AdGroup.RetargetingLists`.
9. Не выводить программистские константы из кода: `KNOWN_GOOD_PACKAGES`, `TARGET_CPA`, `BAN_COST_MULTIPLIER`, `HIGH_CTR` – пишем описание правила на русском, не имя константы.
10. Не выводить хэши и служебные ID: хэши картинок (`4yiz...`), хэши условий, любые служебные ID API. Допустимы только ID объявлений / групп / РК / счётчиков (заказчик найдёт их в Директе).
11. Не давать ссылок на сырые файлы (`data/*.json`, `data/*.tsv`) – заказчик их не открывает (§6.11). Анализ – таблицей внутри отчёта.
12. Не ссылаться на прошлые отчёты / прогоны («в прогоне №1», «как раньше говорили») – самодостаточный отчёт без отсылок наружу (§6.13). Если нужна динамика – повторяем цифру явно.

## Глоссарий обязательных замен (§3.6)

13. UI / UI Директа -> интерфейс / интерфейс Директа.
14. бан-лист / банить -> чистка площадок / отключить площадку / минус-площадки.
15. минусование -> оптимизация.
16. эвристика -> правило / признак.
17. DRAFT -> черновик.
18. регион 213 (любой код региона) -> Москва (название из справочника).
19. Tracking params -> параметры отслеживания / utm-метки.
20. фрод / фрод-кольцо -> подозрительная схема / подозрение на скрутку.
21. «качество трафика упадёт» / «деградация качества трафика» -> «качество результатов РК может упасть».
22. «Сигнал» (про микроцели) -> «оценка параметров вовлечённости» / «поведение на сайте».
23. «клиент» (как адресат отчёта) -> «заказчик» – единственный допустимый термин. У заказчика «клиент» – это покупатель его продукта, путать нельзя.
24. «маркетолог» (как адресат рекомендации) -> «заказчику» (бизнес-решения) или «специалисту по ведению РК» (правки в кабинете). «Маркетолог» – наш внутренний термин, в отчёт не выходит (§6.14).
25. «потолок маркетолога» -> «целевой CPA от заказчика» / «целевой потолок».
26. «псевдо-CPA» / «псевдо-X» – не использовать; либо не приводим число, либо приводим с прямым объяснением, без приставки «псевдо».
27. «фиксированные параметры», «параметризованный запрос», «снапшот» – не выводить, это наш язык (§6.2).
28. «структурный мусор» (про архив/черновики) – не использовать; архив – это нормальная гигиена (§6.10).
29. «Имени РК соответствует» / «не противоречит» – не выводить; норма не озвучивается, озвучиваем только противоречие (§6.4).
30. «Срез проверили, аномалий нет» – не выводить отдельной строкой; писать одной строкой в общем выводе «срезы по X, Y, Z – без значимых отклонений» (§6.8).

## Открытые вопросы и осторожность формулировок (§2, §6.5, §6.7)

31. Открытые вопросы заказчику – выносить ЯВНО. Не угадывать решения за заказчика («вероятно, эти РК нужно удалить» – нельзя).
32. Не предполагать – РАЗБИРАТЬСЯ. «Могут конкурировать» / «отличаются, видимо, текстом» – это работа аудитора, а не заказчика. Сначала сверяем по данным, потом формулируем.
33. 0 корректировок ≠ «не работают». Для свежей РК – «корректировки пока не заданы» (факт), а не ярлык «с этим направлением не работают» (§6.5).

## Финальный grep-чек перед сдачей (§5)

34. Прогнать по `report.md` команды из §5 – не должно быть вхождений: `\bклиент(а|у|ом|...)?\b`, `\bмаркетолог`, ИДЕНТИФИКАТОРОВ_ИЗ_КОДА, хэшей, «псевдо-», «в нашей методологии», «по методологии», «БЗ §», ссылок на `data/*.json`, ссылок на `run-0N` / «прогон №», «аномалий нет / срез проверили».

## Применение

- Любой клиентский отчёт перед отправкой проходит этот чек-лист.
- При написании КП: применять п. 1–3, 7, 31–33 (для заказчика, без жаргона, открытые вопросы – явно, без угадывания за клиента).
- При написании брифов: п. 1–6, 13–30 (терминология) + п. 31–33 (формулировки).
