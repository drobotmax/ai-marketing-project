# KUBRIK - AI-Native Operating Model Notes

Date: 2026-04-04
Context: ideas adapted from `https://t.me/cryptoEssay/2935` and Miras by Improvado (`https://lp.improvado.io/miras-autonomous-business-execution`)

## What matters from the post

The useful idea is not "let's say we are AI-native".
The useful idea is to redesign KUBRIK as an operating system for marketing execution:

- all relevant company knowledge becomes machine-readable
- every task becomes a resumable session
- every repeatable process becomes a maintained skill
- people stop being only executors and become system managers
- each action leaves memory so the system compounds

This fits KUBRIK because we are selling outcome-based service, not software seats.

## Source signals we should borrow

From the Telegram post:
- 100% of information should become tokens accessible to agents
- each task/project should be a session that can be continued, forked, or handed over
- each session should include artifacts, tool use, and next steps
- employees should improve systems and guarantee quality
- each business process should become a visual skill

From the Miras page:
- strong systems combine **eyes, hands, memory**
- the moat is not the model, but the compounding context graph
- the right loop is **Act -> Learn -> Act Better**
- work should happen inside tools the team already uses
- autonomy should be staged: auto-reply -> human-in-the-loop -> full delegation

## Translation for KUBRIK

### 1. KUBRIK is not just a pipeline

Current framing:
- strategist -> fact-checker -> targeting -> media planner -> copywriter -> validator

Better framing:
- KUBRIK is a client operating layer with agents, memory, artifacts, and learnings

The pipeline remains important, but it becomes one execution path inside a broader system.

### 2. Client memory must become explicit

Right now a lot of project truth is split across:
- briefs
- strategies
- chats
- calls
- deck edits
- team memory

We need one durable layer per client:
- `client-memory.md` - facts, constraints, positioning, launch status, known objections
- `learnings.md` - wins, failures, anti-patterns, proven moves
- `ops/sessions/*.md` - logs of important tasks with decisions and next steps

This is the beginning of our practical knowledge graph, even if it starts as markdown before becoming something more formal.

### 3. Every important task should become a session

Examples:
- sales call review
- strategy generation
- launch readiness audit
- weekly optimization review
- creative iteration sprint
- client Q&A / objection handling

Each session should capture:
- goal
- inputs
- artifacts used
- decisions
- next step
- follow-ups for KB or skills

That makes handoff, continuation, and forking possible.

### 4. Every repeated move should become a skill

We already have agent skills, but we should think broader:
- launch-readiness skill
- sales-QA skill
- weekly optimization review skill
- client onboarding skill
- market intelligence update skill
- creative fatigue diagnosis skill

Rule:
- if we do something twice, it should want to become a skill
- if a failure repeats, the fix belongs in a skill or checklist

### 5. KUBRIK needs eyes, hands, memory

#### Eyes
- briefs
- landing pages
- ad account exports
- CRM status
- call notes
- analytics / Metrika / dashboards
- market and competitor data

#### Hands
- generate strategy docs
- generate creatives and plans
- produce QA notes
- create decks and reports
- write launch checklists
- trigger monitoring and digest workflows

#### Memory
- client memory
- session logs
- historical learnings
- platform and market playbooks
- what worked by market / offer / audience / channel

Without all three, we only have "smart prompts".

### 6. Our moat is institutional memory for real estate marketing

The repo already contains valuable knowledge.
What is still missing is the loop that turns delivery into reusable advantage:

1. execute for a developer
2. record decisions and outcomes
3. extract learnings
4. update skills / playbooks
5. deliver faster and better for the next client

This should be described internally as our moat.
Externally, we still sell in client language: speed, control, lower cost, less chaos.

## What to change now

### Immediate changes

1. Add `ops/` to each active client folder
2. Start `client-memory.md` for every real prospect or client
3. Require a session note for any meaningful audit, launch, strategy cycle, or sales review
4. Create a habit that every test ends with a learning update

### Next architecture step

Introduce three operating modes:

1. `Reply`
- fast answers, lookups, summaries

2. `Recommend`
- agent proposes a plan or decision with rationale

3. `Execute`
- agent produces the deliverable or completes the approved task

This maps well to how trust will grow with Anton and Dima.

### Product / sales implication

Do not sell this as "10 agents".
Sell this as:
- a memory-backed marketing operating layer
- every launch becomes faster because the system remembers
- your marketing no longer resets every time a contractor changes

## Suggested language for internal use

Internal:
- "KUBRIK is a memory-backed operating system for real-estate marketing execution."
- "Every client run should improve the system."
- "No orphan work: every meaningful task leaves artifacts and next steps."

External:
- "We don't replace your team with a chatbot. We give you a marketing system that remembers what worked, launches faster, and stays under control."

## 30-day practical rollout

### Week 1
- add `ops/` structure to active clients
- start `client-memory.md` template
- start using session logs for sales QA and strategy work

### Week 2
- add `learnings.md` to active clients
- review recent tests and backfill strongest learnings
- identify 3 repeated workflows that need stronger skills

### Week 3
- connect session outputs to scorecards / reports / deck evidence
- define what counts as `Reply`, `Recommend`, `Execute`

### Week 4
- summarize what operational memory already compounds
- turn the best internal examples into sales material

## Non-goals

- do not turn KUBRIK into a generic agent platform
- do not sell "knowledge graph" jargon to developers
- do not overbuild infra before first paying clients

The right move now is lightweight structure with strong discipline.
